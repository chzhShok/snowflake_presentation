### Слайд 1 — Вводный (~30 сек)

Добрый день. Я расскажу про статью «The Snowflake Elastic Data Warehouse», опубликованную на SIGMOD 2016 — это одна из самых цитируемых работ по архитектуре облачных систем. Авторы описывают, как спроектировали систему с нуля под облако, и какие архитектурные решения при этом приняли. Для нас это интересно именно как кейс принятия архитектурных решений под конкретные нефункциональные требования.

### Слайд 2 — Проблема и контекст (~1:30)

Сначала про контекст — зачем вообще строить что-то новое.

К 2012 году доминировала **shared-nothing архитектура** — каждый узел хранит свой кусок данных и обрабатывает только его. Она хорошо масштабируется, но имеет одну фундаментальную проблему: **вычисления и хранилище жёстко связаны**. Авторы в статье прямо называют три следствия этой связанности.

Первое — **гетерогенность нагрузки**: конфигурация железа, оптимальная для bulk load с высоким I/O, плохо подходит для тяжёлых аналитических запросов с интенсивными вычислениями. Приходится брать компромиссную конфигурацию с низкой средней утилизацией.

Второе — **membership changes**: если нужно добавить или убрать узел, надо перераспределить большие объёмы данных. Причём те же узлы, которые делают решаффлинг, одновременно обрабатывают запросы — система деградирует.

Третье — **online upgrade** практически невозможен, потому что всё связано и предполагается гомогенным.

В облаке, в отличие от on-premise, всё это — не исключения, а норма. Авторы пришли к выводу: нужна архитектура, рождённая под облако, а не портированная в него.

### Слайд 3 — Трёхслойная SOA-архитектура (~1:30)

Ключевое решение — Snowflake построен как **Service-Oriented Architecture** из трёх независимых слоёв, которые общаются через RESTful-интерфейсы. Это не просто логическое разделение — каждый слой деплоится, масштабируется и обновляется независимо.

**Database Storage** — данные в Amazon S3. Принципиальный момент: авторы рассматривали написание собственного хранилища на базе HDFS, но выбрали S3, мотивируя это тем, что удобство использования и надежность S3 «трудно превзойти», а инженерные усилия лучше вложить в кэширование на уровне вычислений.

**Query Processing** — виртуальные хранилища, кластеры Amazon Elastic Compute Cloud (аренда вируталок). «Мышцы» системы.

**Cloud Services** — «мозг»: оптимизатор, менеджер транзакций, метаданные, авторизация. Это **multi-tenant (1 экземпляр обслуживает много клиентов) stateless-сервис (без состояния)**, реплицированный по нескольким availability zones. Данные или сервис копируют и держат синхронно (или почти синхронно) в нескольких зонах доступности (Availability Zones, AZ) внутри одного региона. Отказ отдельного узла не теряет данных — максимум, текущий запрос нужно перезапустить.

Слой, где лежат данные, и слой сервисов (метаданные, оркестрация и т.п.) — общие для всех клиентов и работают постоянно.
Виртуальные склады (warehouses) — отдельные для задач/пользователей и живут недолго: их включают, масштабируют или гасят по необходимости.
У этих частей разный жизненный цикл, и в архитектуре их намеренно разделили: одно — «фундамент для всех», другое — «вычислитель под конкретную нагрузку».

### Слайд 4 — Слой хранения: PAX, иммутабельность, pruning (~2:00)

Теперь подробнее про хранилище — здесь несколько нетривиальных решений.

S3 — это object storage с HTTP PUT/GET/DELETE интерфейсом. Он не поддерживает append и random write: файл можно только перезаписать целиком. Это жёсткое ограничение, которое **напрямую продиктовало весь дизайн формата данных**.

Таблицы разбиваются на **иммутабельные файлы** — аналог блоков в традиционных СУБД. Внутри каждого файла используется формат **PAX (Partition Attributes Across)** — гибридный колонковый: значения каждого атрибута сгруппированы вместе внутри файла. Это позволяет S3 отдавать только нужные колонки через range GET-запросы.

Иммутабельность — не просто следствие ограничений S3. Она делает **MVCC (Multiversion Concurrency Control) тривиальным**: UPDATE создаёт новую версию файла, старая сохраняется. Нет concurrent write conflicts, нет журнала отмены, нет buffer pool (кэш страниц данных в памяти). Транзакции реализованы через **Snapshot Isolation** — все читатели видят consistent snapshot на момент начала транзакции, без блокировок.

Вместо традиционных B+-tree индексов Snowflake использует **min-max pruning**, известный в литературе как «small materialized aggregates» или «zone maps». Для каждого файла хранятся минимум и максимум каждой колонки. При обработке запроса оптимизатор проверяет метаданные и пропускает целые файлы. Это масштабируется, не требует пользовательских усилий и хорошо работает для sequential scan больших данных — именно то, что нужно для аналитики.

Во время join система учится по факту, какие ключи реально есть в одной таблице, и подрезает вторую таблицу, чтобы не обрабатывать заведомо лишние строки — по духу как join с блум-фильтром.

### Слайд 5 — Виртуальные Warehouses: изоляция и elasticity (~1:30)

Виртуальные warehouses — это чистый вычислительный ресурс. Каждая рабочая нода никогда не шарится между двумя хранилищами — это даёт **сильную изоляцию производительности**. Каждый запрос выполняется в отдельном рабочем процессе, который живёт ровно на время запроса — это называется ephemeral process model.

Рабочий процесс никогда не производит наблюдаемые извне сайд эффекты сам по себе — потому что файлы иммутабельны. Это значит, что **обработка отказов тривиальна**: если нода упала, запрос просто перезапускается. Никакого восстановления состояния.

**Consistent hashing** по имени файла закрепляет файл за конкретной нодой: при повторном чтении тот же узел с большей вероятностью отдаст данные из локального кэша, а не тянет их из медленного хранилища — растёт **cache hit rate** (доля запросов, закрытых кэшем). При добавлении или удалении нод система **не делает сразу полную перекладку кэша** (lazy-подход): содержимое постепенно обновляется по мере обращений и вытесняется по LRU (Least Recently Used). Это намеренный компромисс: **стабильность и доступность важнее**, чем мгновенно «идеальная» укладка кэша после каждого изменения кластера.

Ещё одна деталь — **file stealing**: если нода закончила обрабатывать свои файлы раньше других (stragglers), она «крадёт» файлы у перегруженных нод. Причём читает их прямо из S3, а не с перегруженной ноды — дополнительная нагрузка на straggler исключена.

### Слайд 6 — Cloud Services: Cascades-оптимизатор и online upgrade (~1:00)

**Cloud Services** — общий для многих клиентов слой сервисов **без собственного состояния** на нодах: логика живёт там, данные запроса — отдельно. Оптимизатор запросов построен на **Cascades**: план строится **сверху вниз**, варианты перебираются и сравниваются **по оценочной стоимости** (cost-based). Индексы Snowflake **намеренно не использует** — меньше ветвлений при выборе плана и **стабильнее** поведение запросов. Отдельные шаги, например **как размазать join по нодам** (join distribution), решаются **уже во время выполнения**, а не только на этапе компиляции плана — так реже «залипают» заведомо плохие решения оптимизатора.

Отдельно важен паттерн **online upgrade**: новая и старая версии Cloud Services могут работать **параллельно** на одном продакшене. Так можно сделать, потому что сервисы **не хранят долгоживущее состояние внутри процессов** — оно лежит в **транзакционном key-value хранилище**, а код легко подменять. Клиенты переводят на новую версию **постепенно**; уже запущенные запросы спокойно **дорабатывают на старой**, без жёсткого «всех сразу выкинуть». В момент статьи Snowflake выкатывал обновления **раз в неделю** и без остановки доступа к сервису. Это следствие того, что архитектуру **разложили по слоям** и отделили вычисления от хранения состояния.

### Слайд 7 — ELT вместо ETL и execution engine (~1:00)

Традиционный подход — ETL: данные трансформируются до загрузки. Snowflake предлагает **ELT: сначала загрузка, трансформация потом**. Это стало возможным благодаря типу VARIANT — нативная поддержка JSON, Avro, XML без предварительного определения схемы. Snowflake автоматически определяет тип и хранит часто встречающиеся поля в отдельных колонках — статистическим анализом по каждому файлу.

С архитектурной стороны это **развязывает тех, кто пишет данные (producers), и тех, кто их читает (consumers)**: им не нужно одновременно договариваться о каждом изменении схемы. В классическом **ETL** смена схемы — это общий проект на несколько команд и часто месяцы согласований. В **ELT** источник может **просто начать отдавать новые поля** в хранилище, а потребители подхватят их **тогда, когда сами будут готовы** — без общего «релиза схемы» на всех.

Движок исполнения — **columnar** (работает с колонками, а не целыми строками), **vectorized** и **push-based**. **Vectorized execution**: одна и та же операция обрабатывает **пачку из тысяч строк за раз** (SIMD-дружелюбно), при этом **не обязательно материализовать** большие промежуточные таблицы на диск — в отличие от типичного **MapReduce**, где стадии часто пишут и читают объёмы между шагами. **Push-based model** вместо классической схемы **Volcano (pull-based)**: данные **проталкиваются вниз** по дереву операторов сверху вниз, а не «вытягиваются» вызовами `next()` снизу вверх; из **tight loops** (самых горячих циклов) убирается лишняя **control flow logic**, и **cache efficiency** улучшается — массивы обрабатываются более предсказуемо. По духу исполнения авторы отсылают к **MonetDB/X100**.

### Слайд 8 — Выводы для архитектора ПО (~1:00)

Какие уроки можно вынести с точки зрения проектирования ПО?

**Первое: ограничения платформы диктуют дизайн**. Иммутабельные файлы появились не из любви к иммутабельности, а потому что S3 не поддерживает append. Из иммутабельности вытекли MVCC, тривиальная обработка отказов и lazy consistent hashing. Одно техническое ограничение развернуло весь дизайн данных.

**Второе: разделение ответственностей как физический, а не логический принцип**. Каждый слой — отдельный сервис со своим lifecycle, масштабированием и командой. Это позволяет релизиться еженедельно без даунтайма. Знакомый принцип Single Responsibility, но применённый к инфраструктуре.

**Третье: осознанные trade-offs**. Lazy consistent hashing — сознательный выбор доступности над оптимальностью кэша. Отказ от индексов — выбор предсказуемости над пиковой производительностью. Каждое решение в статье объясняется через trade-off, а не через «это лучший способ».

**Четвёртое: влияние на индустрию**. После Snowflake модель «shared storage + independent compute» скопировали BigQuery, Redshift Serverless, Azure Synapse. Это тот случай, когда одна архитектурная идея переформатировала целую категорию систем.
